### HTML과 HTTP 통신: 개념 및 동작 원리

---

## **HTML (HyperText Markup Language)**

### **HTML의 개념**
- HTML은 **웹 페이지의 구조를 정의**하는 마크업 언어로, 웹 브라우저가 이를 해석하여 화면에 콘텐츠를 표시합니다.
- "하이퍼텍스트"는 문서 간의 링크를 의미하며, HTML은 텍스트, 이미지, 비디오 등 다양한 요소를 포함하여 웹 페이지를 구성합니다.

### **HTML의 주요 특징**
1. **구조적 정의**:
   - HTML은 태그(`<tag>`)를 사용하여 웹 페이지의 구조와 내용을 정의합니다.
   - 태그는 여는 태그(`<tag>`)와 닫는 태그(`</tag>`)로 구성되며, 일부 태그는 닫는 태그가 없습니다(예: `<img>`).

2. **플랫폼 독립성**:
   - HTML 문서는 운영체제와 상관없이 모든 웹 브라우저에서 동일하게 작동합니다.

3. **계층적 구조**:
   - HTML은 중첩된 태그로 계층적 구조를 형성하며, 이를 통해 콘텐츠의 논리적 관계를 정의합니다.

4. **다양한 멀티미디어 지원**:
   - 이미지(`<img>`), 오디오(`<audio>`), 비디오(`<video>`) 등의 태그를 통해 멀티미디어 콘텐츠를 포함할 수 있습니다.

5. **폼(Form) 요소**:
   - `<form>` 태그를 사용하여 사용자 입력을 받을 수 있으며, 텍스트 입력, 체크박스, 라디오 버튼 등 다양한 입력 방식을 지원합니다.

### **HTML 기본 구조**
```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>
<body>
    <h1>Hello World</h1>
    <p>This is a paragraph.</p>
    <a href="https://example.com">Visit Example</a>
</body>
</html>
```

### HTTP (HyperText Transfer Protocol): 개념, 동작 원리 및 특징

---

## **HTTP의 개념**

- HTTP는 **클라이언트(웹 브라우저)**와 **서버** 간 데이터를 주고받기 위한 애플리케이션 계층 프로토콜입니다.
- 주로 HTML 문서, 이미지, CSS, JavaScript 등 웹 리소스를 전송하는 데 사용됩니다.
- TCP/IP를 기반으로 동작하며, 기본적으로 80번 포트를 사용(HTTPS는 443번 포트).

---

## **HTTP의 주요 특징**

1. **비연결성(Connectionless)**:
   - 요청(Request)과 응답(Response)이 완료되면 클라이언트와 서버 간의 연결이 끊어집니다.
   - 새로운 요청이 발생하면 새로운 연결을 생성합니다.

2. **무상태성(Stateless)**:
   - 각 요청은 독립적으로 처리되며, 서버는 클라이언트의 이전 요청 상태를 기억하지 않습니다.
   - 상태 유지를 위해 쿠키, 세션 또는 토큰 기반 인증이 필요합니다.

3. **확장성(Extensibility)**:
   - HTTP 헤더를 통해 추가 정보를 전달할 수 있어 유연성과 확장성이 높습니다.
   - 예: `Content-Type`, `Authorization`, `Cache-Control` 등 다양한 헤더 사용 가능.

4. **텍스트 기반 프로토콜**:
   - 요청과 응답 메시지가 사람이 읽을 수 있는 텍스트 형식으로 작성됩니다.

---

## **HTTP의 동작 원리**

### 1. 클라이언트 요청(Request)
- 클라이언트(브라우저)가 서버에 리소스를 요청하는 메시지를 생성합니다.
- 요청 메시지는 다음과 같은 요소로 구성됩니다:
  1. **요청 라인(Request Line)**:
     - HTTP 메서드(GET, POST 등), 요청 URI, HTTP 버전 포함.
     ```http
     GET /index.html HTTP/1.1
     ```
  2. **헤더(Header)**:
     - 요청에 대한 메타데이터를 포함(예: `User-Agent`, `Accept`, `Host`).
     ```http
     Host: www.example.com
     User-Agent: Mozilla/5.0
     ```
  3. **본문(Body)**:
     - POST, PUT 메서드에서 전송되는 데이터(예: 폼 데이터).

### 2. 서버 응답(Response)
- 서버는 클라이언트의 요청을 처리한 후 응답 메시지를 반환합니다.
- 응답 메시지는 다음과 같은 요소로 구성됩니다:
  1. **상태 라인(Status Line)**:
     - HTTP 버전, 상태 코드(200, 404 등), 상태 메시지 포함.
     ```http
     HTTP/1.1 200 OK
     ```
  2. **헤더(Header)**:
     - 응답에 대한 메타데이터를 포함(예: `Content-Type`, `Content-Length`).
     ```http
     Content-Type: text/html; charset=UTF-8
     Content-Length: 1234
     ```
  3. **본문(Body)**:
     - HTML 문서나 JSON 데이터 등 실제 데이터.

## **HTTP 메서드**

HTTP 메서드는 클라이언트가 서버에 어떤 작업을 요청하는지 정의합니다.

| 메서드   | 설명                              | 사용 예시                             |
|----------|-----------------------------------|---------------------------------------|
| GET      | 리소스 조회                      | 웹 페이지 로드                        |
| POST     | 데이터 전송 및 리소스 생성        | 회원가입, 로그인                      |
| PUT      | 리소스 전체 수정                 | 사용자 정보 수정                      |
| DELETE   | 리소스 삭제                      | 게시물 삭제                           |
| PATCH    | 리소스 일부 수정                 | 비밀번호 변경                         |
| HEAD     | 리소스 헤더 정보만 조회          | 리소스 존재 여부 확인                 |
| OPTIONS  | 지원 가능한 HTTP 메서드 조회      | 서버가 지원하는 메서드 확인           |

---

## **HTTP 상태 코드**

HTTP 상태 코드는 서버가 클라이언트 요청을 처리한 결과를 나타냅니다.

### 주요 상태 코드 분류

| 상태 코드 범위 | 설명                              |
|----------------|-----------------------------------|
| **1xx**        | 정보 전달 (처리 중)               |
| **2xx**        | 성공 (요청 성공)                  |
| **3xx**        | 리다이렉션 (다른 URL로 이동 필요) |
| **4xx**        | 클라이언트 오류                   |
| **5xx**        | 서버 오류                         |

### 자주 사용되는 상태 코드

| 상태 코드 | 설명                              |
|-----------|-----------------------------------|
| 200       | 성공                              |
| 301       | 영구 이동                         |
| 302       | 임시 이동                         |
| 400       | 잘못된 요청                       |
| 401       | 인증 필요                         |
| 403       | 접근 금지                         |
| 404       | 리소스를 찾을 수 없음             |
| 500       | 서버 내부 오류                    |

## **HTTP 버전**

### HTTP/1.0
- 초기 버전으로, 요청마다 새로운 TCP 연결을 생성.
- 비효율적인 네트워크 자원 사용.

### HTTP/1.1
- 기본적으로 Keep-Alive를 지원하여 TCP 연결 재사용 가능.
- 파이프라이닝(Pipelining)을 통해 여러 요청을 동시에 처리 가능.

### HTTP/2
- 바이너리 프로토콜로 전환하여 성능 개선.
- 멀티플렉싱(Multiplexing)을 통해 단일 연결에서 다중 스트림 처리 가능.
- 헤더 압축을 통해 네트워크 오버헤드 감소.

### HTTP/3
- QUIC 프로토콜 기반으로 작동하여 지연 시간 감소.
- UDP를 사용하여 빠른 연결 설정 및 데이터 전송 가능.

---

## **HTTP Keep-Alive**

### 개념
- Keep-Alive는 하나의 TCP 연결을 여러 HTTP 요청/응답에 재사용하는 기능입니다.
- 기본적으로 HTTP/1.1부터 활성화되어 있습니다.

### 장점
1. TCP 연결 설정 및 종료에 필요한 오버헤드 감소.
2. 네트워크 지연 시간 감소로 성능 향상.
3. 대규모 리소스 로드 시 효율적(예: 이미지, CSS, JS 파일).

### 단점
1. 오래된 연결이 많아지면 서버 자원 부족 문제 발생 가능.
2. 비효율적인 연결 유지 시 성능 저하.


---

## 요약

- HTML은 웹 페이지의 구조와 내용을 정의하는 마크업 언어입니다.
- HTTP는 클라이언트와 서버 간 데이터를 주고받기 위한 비연결성, 무상태성을 가진 텍스트 기반 프로토콜입니다.
- 주요 요소는 요청(Request)와 응답(Response), 그리고 이를 정의하는 메서드(GET, POST 등)와 상태 코드(200, 404 등)입니다.
- 최신 버전인 HTTP/3은 UDP 기반으로 성능과 안정성을 더욱 향상시켰습니다.
- 두 기술은 함께 작동하여 웹 페이지를 사용자에게 제공하며, 클라이언트가 HTML 문서를 요청하면 서버가 이를 반환하고 브라우저가 이를 렌더링하여 화면에 표시합니다.
